chore: initialize production architecture skeleton

Initialize the first production-ready project foundation for the Tauri desktop AI dubbing application.

This commit establishes the base monorepo structure, frontend architecture, Rust workspace boundaries, documentation, task orchestration, quality checks, and initial CI pipeline. The goal is to create a scalable foundation before implementing the full YouTube ingest, subtitle processing, AI voice pipeline, media assembly, and export flow.

Main changes:

* Initialize monorepo structure
  * Add `apps/desktop` for the React/Vite desktop frontend
  * Add `src-tauri` as the thin Tauri shell layer
  * Add `crates/` for modular Rust workspace packages
  * Add `docs/` for architecture, conventions, CI, and Taskfile documentation
  * Add `taskfiles/` for separated Taskfile modules
  * Add `tools/scripts/` for repository quality checks
  * Add `.github/workflows/` for CI configuration

* Establish frontend architecture
  * Initialize React + Vite + TypeScript frontend
  * Add Feature-Sliced Design folder structure
  * Add base `app`, `pages`, `widgets`, `features`, `entities`, and `shared` layers
  * Keep pages as composition-only entry points
  * Add initial page/widget/feature skeletons
  * Add Tailwind base styling and centralized theme tokens
  * Prepare structure for reusable UI components, assets, hooks, API wrappers, and config

* Establish Rust workspace architecture
  * Add root Cargo workspace configuration
  * Add modular crates for:
    * `app-shell`, `application`, `domain`, `ports`
    * `adapters-tauri`, `adapters-storage`, `adapters-ytdlp`, `adapters-ffmpeg`, `adapters-model`
    * `jobs`, `common`
  * Add initial domain boundaries for project, transcript, dubbing, media, and job concepts
  * Add initial ports for external integrations and persistence
  * Keep domain code independent from Tauri and infrastructure dependencies
  * Prepare architecture for ports-and-adapters style implementation

* Keep Tauri shell thin
  * Define `src-tauri` as bootstrap-only application layer
  * Keep command registration, state setup, plugin setup, and permissions inside the shell
  * Avoid placing business logic directly in `main.rs`
  * Prepare bridge between React UI and Rust application use cases
  * Add initial health-check style command for smoke testing the frontend-to-Rust path

* Add architecture documentation
  * Add main documentation entry point in `docs/README.md`
  * Document final stack decision: React + Vite + Tauri v2
  * Document high-level application architecture
  * Document frontend Feature-Sliced Design rules
  * Document Rust workspace and ports/adapters boundaries
  * Document CI quality gates
  * Document Taskfile command structure
  * Describe the intended media/AI pipeline

* Add Taskfile orchestration
  * Add root `Taskfile.yml`
  * Add separate task modules for frontend, Rust, docs, and quality checks
  * Add unified local commands (`install`, `dev`, `check`, `ci`)
  * Add frontend tasks for linting, type checking, tests, builds, and Storybook builds
  * Add Rust tasks for workspace check, formatting, Clippy, and tests
  * Make Taskfile the single source of truth for local and CI checks

* Add repository quality scripts
  * Add file-size validation script
  * Add raw color usage validation script
  * Add required documentation validation script
  * Prepare enforcement for small pages, controlled widget/feature size, no raw color values, and clean separation between frontend layers

* Add initial CI pipeline
  * Add GitHub Actions workflow for pull requests and pushes
  * Configure Node, pnpm, Rust, rustfmt, Clippy, and Task
  * Run dependency installation through `task install`
  * Run all quality checks through `task check`
  * Avoid duplicating project logic directly inside CI

* Add onboarding documentation
  * Add development commands to README
  * Document project stack and architecture entry points
  * Explain that the project uses a thin Tauri shell, Rust orchestration, and sidecar binaries

Beyond the original tasks:

* Smart CI Path Filtering
  * Refactor `.github/workflows/ci.yml` using `dorny/paths-filter` to skip irrelevant checks and massively optimize CI execution time
  * Split monolithic `task check` into precise scoped targets (`check:frontend`, `check:rust`, `check:quality:frontend`, etc.)
  * Setup a robust `ci-summary` job pattern that handles skipped jobs natively, allowing it to be the sole branch protection requirement

* Release & Signing Workflows
  * Extract cross-platform production builds to a dedicated `.github/workflows/release.yml` triggered on tags
  * Configure `.github/workflows/tauri-build.yml` for manual smoke testing of builds without generating releases
  * Document exact requirements for OS code signing (Apple ID, Windows EV) vs Tauri auto-updater cryptographic signatures

* Branding & Assets
  * Add core project logo to `apps/desktop/public/logo.jpg`
  * Auto-generate production Tauri icons (`.ico`, `.icns`, `.png`) across all OS and mobile targets using Tauri CLI

* Extra Quality & Security Gates
  * Add Node dependency vulnerability scanning (`pnpm audit`) to the quality gate
  * Mandate English as the exclusive language for all documentation and translate all newly created docs
  * Format all Rust code correctly across the workspace

Architecture decisions introduced:
* React + Vite is the final frontend choice
* Tauri is used only as the desktop shell and secure command bridge
* Rust workspace owns orchestration and domain/application logic
* Heavy external processing should be implemented through controlled sidecar binaries
* Frontend must follow Feature-Sliced Design
* Pages must only compose widgets/features and must not contain business logic
* Design tokens must be centralized through theme files
* Raw colors are not allowed in normal UI implementation
* CI must call Taskfile commands instead of duplicating command logic
* Documentation is treated as part of the quality gate
